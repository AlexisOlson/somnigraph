## 1. Architecture Overview

**Stack**: Python 3.10+, GitPython, OpenAI SDK (via OpenRouter), FastAPI (optional server). ~3,400 lines of Python.
**Stars**: New repo (~21 commits), alpha/prototype status.
**License**: MIT.
**Default LLM**: Gemini 2.5 Pro (writer), Grok 4.1 Fast (retrieval agent). Configurable via env.

DiffMem is a git-native memory backend where the entire storage, indexing, and versioning layer is a Git repository. There are no vectors, no embeddings, no BM25, no database at all. Memory files are Markdown, organized by type (people, contexts, timeline). Retrieval is performed by a multi-turn LLM agent that explores the repository using whitelisted shell commands (cat, grep, ls, head, tail, wc, git log/diff/blame/show/rev-list).

The system is in production powering "Annabelle," a WhatsApp/Messenger bot maintained by Growth Kinetics.

**Module organization**:
- `api.py` -- Clean `DiffMemory` class wrapping read/write operations
- `repo_manager.py` -- Federated git architecture (orphan branches, worktrees, GitHub sync)
- `server.py` -- FastAPI REST server with auth, periodic sync, post-commit webhooks
- `retrieval_agent/` -- Multi-turn LLM agent with sandboxed shell commands
- `writer_agent/` -- LLM-driven entity extraction, creation, update, and indexing

## 2. Memory Model

### Memory types (inspired by SMS model)
- **Biographical** (`memories/people/*.md`): Long-lived identity and relationship profiles. One file per person.
- **Factual/Semantic** (`memories/contexts/*.md`): Themes, projects, organizations. Slowly evolving.
- **Episodic** (`timeline/YYYY-MM.md`): Monthly chronological entries with daily sections.

### File organization (per user)
```
<worktree_root>/
  index.md               -- Autogenerated entity manifest (JSON descriptors)
  {user_id}.md            -- Core user profile
  memories/people/        -- Person entity files
  memories/contexts/      -- Theme entity files
  timeline/               -- Monthly timeline files
```

### Strength tags and plasticity
Files contain `[Strength: High|Medium|Low]` tags on individual blocks (delimited by `### Header` to `/END`). Strength is computed from git statistics -- edit frequency and recency -- during index building (`build_index.txt` prompt). This simulates synaptic plasticity: frequently edited memories are "stronger."

Special block markers control retrieval behavior:
- `[ALWAYS_LOAD]` -- Core identity blocks loaded as a baseline safety net
- `[MERGE_LOAD]` -- Monthly summaries for timeline files

### Git structure: Federated orphan branches
Each user gets an isolated orphan branch (`user/{user_id}`) with zero shared history. Active users are mounted as Git worktrees from a central storage repo (`repo_manager.py`). This provides:
- Complete isolation between users (no cross-contamination)
- Efficient multi-user scaling (shared object store, separate histories)
- GitHub sync for durability (periodic push per branch)
- Clean "Right to be Forgotten" via branch deletion (`wipe_user()`)

## 3. Retrieval Pipeline

### Phase 0: Baseline (deterministic, zero LLM cost)
`baseline.py` loads two things unconditionally:
1. **User entity file** (`{user_id}.md`) -- the core profile
2. **Recent timeline** -- last 30 days of `timeline/YYYY-MM.md` files (max 5 files, 1000 chars each)

Token estimation is `len(text) // 4` throughout.

### Phase 1: Agent exploration (multi-turn LLM)
`agent.py` runs a multi-turn conversation with an LLM (default: Grok 4.1 Fast via OpenRouter). The agent has a single tool: `run(command="...")` which executes whitelisted shell commands in the user's worktree (`command_router.py`).

**Whitelisted commands**: cat, head, tail, grep, ls, wc, git (log/diff/blame/show/rev-list/shortlog only).

**Protocol** (from `retrieval_agent/prompts/system.txt`):
1. **Turn 1 -- Orient**: `cat index.md` to see all entities and their cues
2. **Turn 2 -- Temporal probe**: `git log --format='%h %ad' --date=relative --name-only -15` to see recent changes
3. **Turn 3 -- Targeted investigation**: grep, head, git diff on specific files
4. **Turn 4 -- Prescribe**: Output a JSON retrieval plan with ContentPointers

The agent outputs a structured JSON plan specifying what content to load, with four pointer types:
- `file_section` (line ranges from grep -n)
- `git_diff` (recent changes to a file)
- `git_log` (commit history/messages)
- `file` (entire file, discouraged for large files)

Each pointer has `priority` ("must_include" or "if_budget_allows") and `est_tokens`.

Default config: max 6 turns, 120-second timeout, 20,000 token budget.

### Phase 2: Pointer resolution
`resolver.py` takes the agent's plan and resolves each ContentPointer to actual content. Must-include pointers are processed first, then optional ones. Resolution stops at the token budget. Truncation is applied if a pointer exceeds remaining budget.

### Phase 3: ALWAYS_LOAD safety net
After the agent identifies relevant entities, `load_always_load_for_entities()` loads `[ALWAYS_LOAD]` blocks from those specific entity files. This ensures core identity blocks are present even if the agent's pointers missed them.

### Command sandboxing
`command_router.py` implements a two-layer execution/presentation architecture:
- **Layer 1**: Validates commands against whitelist, parses chains (`|`, `&&`, `||`, `;`), executes via subprocess
- **Layer 2**: Truncates output (150 lines / 30KB max), adds metadata footer, handles binary detection
- Windows compatibility: maps Unix commands to Git for Windows tools, converts single quotes to double quotes

## 4. Write Path

### Session processing (`writer_agent/agent.py`)
The `WriterAgent` orchestrates memory updates through an LLM pipeline:

1. **Semantic index loading**: Reads JSON descriptors appended to each entity file (last line) for fast entity lookup
2. **Entity identification** (`1_identify_entities.txt`): LLM analyzes the session transcript against the semantic index to decide which entities to create vs. update. Explicit noise filtering: "skip single-mention ephemera."
3. **Parallel entity creation/update**: Uses `ThreadPoolExecutor` (configurable, default 8 concurrent LLM calls) to create new entity files and update existing ones simultaneously
4. **Timeline entry creation**: Adds chronological record to the current month's timeline file
5. **Index rebuilding**: Regenerates `index.md` and per-entity JSON semantic descriptors
6. **Git commit**: Atomic commit of all staged changes with descriptive message

Entity creation uses example files as templates -- the LLM sees a well-evolved entity and creates a new one following the same structure. Entity updates receive the existing file content and the new session transcript, producing updated markdown.

### Per-entity semantic index
Each entity file gets a JSON descriptor appended (built by `build_index.txt` prompt):
```json
{"name":"Nova","type":"human","strength":"Medium","hard_cues":["keyword1"],"soft_cues":["mood"],"emotional_cues":["trust"],"related_entities":["slug1"]}
```
These are loaded at retrieval time for the agent to triage entities without reading full files. The master `index.md` aggregates paths and summaries.

### Onboarding
`onboarding_agent.py` creates the full directory structure, user entity file, initial entities, timeline entry, and semantic indexes from a raw information dump or pre-filled template. Inherits from `WriterAgent`.

## 5. Comparison to Somnigraph

### What DiffMem has that we don't

1. **Agentic retrieval with temporal reasoning**: The retrieval agent can ask "how did this entity change recently?" via `git diff` and `git log`. Somnigraph retrieves memories as atomic units; it has no mechanism for surfacing *how* a memory evolved over time. The agent can follow chains of reasoning across multiple exploration turns -- e.g., discovering entity co-occurrence in commit history, then diving into specific diffs.

2. **Human-readable, editable storage**: Memory files are plain Markdown that users can read and edit directly. Somnigraph's SQLite database is opaque without tooling.

3. **Current-state focus with full history**: Entity files maintain a compact "now" view while Git preserves every past state. This separation keeps the query surface small. Somnigraph stores all versions as separate memories with decay, which can bloat the candidate pool.

4. **Per-entity semantic descriptors**: The 160-token JSON index appended to each file (with hard/soft/emotional cues) gives the retrieval agent instant triage capability without loading full files. Somnigraph uses theme tokens and embeddings for similar purposes but doesn't have a structured cue taxonomy.

5. **Federated multi-user architecture**: Orphan branches provide cryptographic-strength user isolation with clean deletion. Somnigraph is single-user by design.

6. **Production deployment with real users**: Annabelle is processing real WhatsApp/Messenger conversations. While Somnigraph is also in production (for one user), DiffMem's multi-user server architecture is more mature.

### What we have that they don't

1. **Quantitative retrieval scoring**: Somnigraph has a 26-feature learned LightGBM reranker trained on 1032 queries with measured NDCG. DiffMem's retrieval quality depends entirely on the LLM agent's judgment -- there's no scoring, no feedback loop, no way to measure or improve retrieval precision systematically.

2. **Feedback loop**: `recall_feedback()` creates a closed loop where retrieval quality improves over time. DiffMem has no feedback mechanism.

3. **Hybrid multi-channel retrieval**: BM25 + vector + theme overlap + PPR graph expansion, fused with RRF. DiffMem uses only what the LLM agent can find via shell commands -- powerful for temporal reasoning but limited for semantic similarity.

4. **Graph structure**: Typed edges (support, contradict, evolve, derive) with Hebbian co-retrieval strengthening and PPR expansion. DiffMem has markdown links between files but no graph traversal at retrieval time.

5. **Biological decay model**: Per-memory exponential decay with category-specific defaults and feedback-modulated adjustment. DiffMem's Strength tags are static labels derived from edit frequency, not a continuous decay function.

6. **Sleep consolidation**: Three-phase offline pipeline (NREM classification, REM LLM-mediated decisions, archiving). DiffMem mentions pruning and archiving as future features in repo_guide.md but has no implementation.

7. **Embeddings and vector search**: Somnigraph uses OpenAI text-embedding-3-small for semantic similarity. DiffMem deliberately avoids all vector/embedding infrastructure.

8. **Evaluation infrastructure**: Ground truth pipeline, LLM-as-judge, LoCoMo benchmark (85.1% overall accuracy). DiffMem has a test_context.py script for A/B comparison but no systematic evaluation.

### Architectural trade-offs

| Dimension | DiffMem | Somnigraph |
|-----------|---------|------------|
| Storage | Git + Markdown files | SQLite + sqlite-vec + FTS5 |
| Retrieval method | Multi-turn LLM agent with shell commands | Hybrid BM25/vector/theme + learned reranker |
| Temporal reasoning | Native (git diff/log/blame) | Weak (decay + timestamps, no diff capability) |
| Latency per query | High (multiple LLM calls, ~2-30s) | Low (single DB query + model inference, ~200ms) |
| Cost per query | Multiple LLM API calls for retrieval | Zero LLM cost for retrieval |
| Editability | Direct markdown editing | Requires API/tools |
| User isolation | Git orphan branches | Single-user |
| Retrieval quality measurement | None (LLM judgment) | Learned reranker, NDCG, feedback loop |
| Dependencies | gitpython, openai | sqlite, lightgbm, openai (embeddings only) |
| Offline capability | Full (git is local) | Full (SQLite is local) |

## 6. Worth Adopting?

**Temporal diff surfacing**: The core insight -- that git diffs and commit history encode temporal evolution -- is genuinely novel in the memory system space. Somnigraph could benefit from tracking how memories change over time, not just storing current state. However, implementing this wouldn't require git; it could be done with a `memory_versions` table in SQLite.

**Structured entity descriptors**: The per-entity JSON index with hard/soft/emotional cues is a compact, useful abstraction. Somnigraph's theme tokens serve a similar role but without the structured taxonomy. Worth considering as a richer metadata schema.

**Nothing else is directly adoptable**. The agentic retrieval approach is interesting but fundamentally trades latency and cost for flexibility. Somnigraph's learned reranker achieves sub-200ms retrieval with measurable quality. Adding multiple LLM calls per query would be a regression.

## 7. Worth Watching

- **Retrieval quality metrics**: DiffMem's roadmap mentions LoCoMo evaluation. If they benchmark the agentic retrieval approach, the results would be informative -- it would tell us whether multi-turn LLM exploration outperforms traditional hybrid retrieval on standardized QA.
- **Agent-driven pruning**: The repo_guide.md describes future "smart forgetting" via archiving to git branches. If implemented, the approach to deciding what to prune would be interesting to compare against Somnigraph's sleep consolidation.
- **Visual retrieval for context compression**: Listed on the roadmap. Unclear what this means, but worth monitoring.
- **Scale behavior**: How does the system perform with hundreds of entity files per user? The agent currently reads `index.md` for triage, which could become unwieldy.

## 9. Relevance to Somnigraph

**Medium**. DiffMem represents a genuinely different architectural philosophy -- using git as both storage and retrieval index, with an LLM agent replacing traditional search infrastructure. The temporal reasoning capability (surfacing diffs and evolution patterns) is the most interesting aspect and highlights a gap in Somnigraph's approach. However, the fundamental trade-off (high latency + high cost per query vs. measurable retrieval quality) makes direct adoption of the pattern impractical for Somnigraph. The system is better understood as a proof-of-concept that agentic exploration of structured text can substitute for traditional IR pipelines, with the open question being whether it can match or exceed their precision at acceptable cost.
