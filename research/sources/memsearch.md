# memsearch — Source Analysis

*Phase 13, 2026-03-05. Analysis of zilliztech/memsearch (v0.1.15 library, v0.2.2 plugin).*

## 1. Architecture Overview

**Language:** Python 3.10+, shell scripts (bash) for Claude Code hooks.

**Key dependencies:** pymilvus, milvus-lite (non-Windows), openai, watchdog, click, tomli_w. Optional: google-genai, voyageai, ollama, sentence-transformers, anthropic.

**Storage layout:**
```
.memsearch/                    # Project-scoped state
  memory/                      # Markdown memory files (source of truth)
    2026-02-05.md              # Daily logs
    2026-02-06.md
  .watch.pid                   # Singleton watcher PID file
  milvus.db                    # Milvus Lite local database (default)
~/.memsearch/
  config.toml                  # Global config
  milvus.db                    # Default Milvus Lite DB location
.memsearch.toml                # Per-project config
```

**Source structure:**
```
src/memsearch/
  core.py          # MemSearch orchestrator class (index, search, compact, watch)
  store.py         # MilvusStore: Milvus wrapper with hybrid search
  chunker.py       # Markdown heading-based chunking + SHA-256 dedup
  scanner.py       # Recursive .md file discovery
  config.py        # Layered TOML config system
  cli.py           # Click CLI (index, search, watch, compact, expand, transcript, stats, reset, config)
  watcher.py       # watchdog-based file watcher with debounce
  compact.py       # LLM-powered chunk summarization
  transcript.py    # JSONL transcript parser for Claude Code sessions
  embeddings/
    __init__.py    # Protocol + lazy-loading factory
    openai.py      # text-embedding-3-small (1536d), batch 2048
    google.py      # gemini-embedding-001
    voyage.py      # voyage-3-lite
    ollama.py      # nomic-embed-text
    local.py       # all-MiniLM-L6-v2 (sentence-transformers)
ccplugin/
  .claude-plugin/plugin.json
  hooks/           # 4 lifecycle hooks (bash)
  scripts/         # derive-collection.sh
  skills/          # memory-recall skill (SKILL.md)
```

## 2. Markdown-First Storage

**Core philosophy:** Markdown is the single source of truth. The vector index is a derived cache, rebuildable from `.md` files at any time.

**File format:** One file per day, named `YYYY-MM-DD.md`. Created by SessionStart hook, appended to by Stop hook.

```markdown
## Session 14:30

### 14:30
<!-- session:abc123def turn:ghi789jkl transcript:/path/to/session.jsonl -->
- Implemented caching system with Redis L1 and in-process LRU L2
- Fixed N+1 query issue in order-service using selectinload

## Session 17:45

### 17:45
<!-- session:mno456pqr turn:stu012vwx transcript:/path/to/session.jsonl -->
- Debugged React hydration mismatch caused by Date.now() during SSR
```

**Human editability:** Fully editable. Files are plain markdown with optional HTML comment anchors. Users can add, edit, or delete memory entries directly. The watcher or a manual `memsearch index` picks up changes.

**Anchor format:** HTML comments (`<!-- session:UUID turn:UUID transcript:PATH -->`) embedded in markdown enable progressive disclosure drill-down to original transcripts. These are metadata, not structural.

**How the vector index derives from it:** `scan_paths()` walks directories for `.md` files, `chunk_markdown()` splits by headings (with paragraph-boundary fallback for oversized sections), each chunk gets a SHA-256 content hash. `compute_chunk_id()` creates a composite ID: `hash(markdown:source:startLine:endLine:contentHash:model)`. Only chunks with new IDs are embedded and upserted. Stale chunks (from edited/deleted content) are garbage-collected on every index run.

## 3. Vector Index

**Embedding model:** Pluggable. Default is OpenAI `text-embedding-3-small` (1536 dimensions). Five providers: openai, google (gemini-embedding-001), voyage (voyage-3-lite), ollama (nomic-embed-text), local (all-MiniLM-L6-v2 via sentence-transformers).

**Index construction:**
- Milvus collection schema has 8 fields: `chunk_hash` (VARCHAR PK), `embedding` (FLOAT_VECTOR), `content` (VARCHAR with BM25 analyzer), `sparse_vector` (auto-generated by BM25 Function), `source`, `heading`, `heading_level`, `start_line`, `end_line`.
- Two indexes: FLAT on dense embedding (COSINE metric), SPARSE_INVERTED_INDEX on sparse_vector (BM25 metric).
- Dedup via composite chunk ID as primary key. SHA-256 hash of `source:startLine:endLine:contentHash:model` — so re-embedding with a different model creates new entries.

**Search:** Hybrid search using `pymilvus.hybrid_search()` with two `AnnSearchRequest`s (dense cosine + BM25 sparse), fused with `RRFRanker(k=60)`. This is real RRF, not a custom implementation.

**Rebuild capability:** Yes, trivially. `memsearch reset --yes` drops the collection, then `memsearch index .memsearch/memory/ --force` rebuilds everything from markdown. The `--force` flag skips dedup checks and re-embeds all chunks.

**Milvus backends:** Three modes — Milvus Lite (local `.db` file, no Windows support), Milvus Server (HTTP), Zilliz Cloud (HTTPS). The plugin auto-detects the backend from the URI and adjusts behavior (Lite gets one-time index, Server/Cloud get persistent watcher).

## 4. Plugin Integration

**Integration type:** Claude Code **plugin** (NOT MCP). Uses the native plugin system with `hooks.json`, `plugin.json`, and `SKILL.md`.

**Plugin registration:**
- Marketplace: `marketplace.json` at repo root
- Install: `/plugin marketplace add zilliztech/memsearch` then `/plugin install memsearch`
- Dev mode: `claude --plugin-dir ./ccplugin`

**Hooks used (4 lifecycle hooks):**

| Hook | Trigger | Async | Timeout | Purpose |
|------|---------|-------|---------|---------|
| SessionStart | Session begins | No | 10s | Bootstrap memsearch (auto-install via uvx if needed), start watcher, write session heading, inject cold-start context (last 30 lines from 2 most recent daily logs), display status |
| UserPromptSubmit | Each user prompt | No | 15s | Lightweight `systemMessage` hint: "[memsearch] Memory available" (skips prompts <10 chars) |
| Stop | After each Claude response | **Yes** | 120s | Extract last turn from transcript, summarize with `claude -p --model haiku`, append to daily `.md` with session anchors, re-index |
| SessionEnd | Session exits | No | 10s | Stop watcher process, cleanup PID file |

**Skill (1):**
- `memory-recall` — `context: fork` (isolated subagent). Allowed tools: Bash only. Claude auto-invokes when it judges context is needed. The subagent runs `memsearch search`, evaluates, `memsearch expand`, optionally `memsearch transcript`, returns curated summary. Main context window only sees the final summary.

**Key design decision:** No MCP server, no sidecar process, no Express worker. Everything is shell scripts calling the `memsearch` CLI + a skill definition. The forked subagent pattern keeps retrieval costs out of the main context window.

**Per-project isolation:** `derive-collection.sh` creates a unique Milvus collection name per project: `ms_<sanitized_basename>_<8char_sha256_of_path>`.

## 5. Auto-Session Summary

**Trigger:** The `Stop` hook fires after each Claude response (async, non-blocking).

**Process:**
1. Recursion guard: checks `stop_hook_active` flag to prevent infinite loops (since the hook calls `claude -p` internally).
2. Validates transcript (must exist, >= 3 lines).
3. `parse-transcript.sh` extracts only the **last turn** — finds the last real user message and everything after it to EOF. Formats with role labels (`[Human]`, `[Claude Code]`, `[Claude Code calls tool]`, `[Tool output]`/`[Tool error]`). Tool results truncated to 1000 chars.
4. Pipes to `claude -p --model haiku --no-session-persistence` with a detailed third-person note-taker system prompt requesting 2-6 bullet points.
5. Output appended to `MEMORY_DIR/YYYY-MM-DD.md` under a `### HH:MM` sub-heading with HTML comment anchor containing session ID, turn UUID, and transcript path.
6. Runs `memsearch index` immediately (doesn't rely on watcher).

**Key detail:** Only the last turn is summarized per stop event, not the entire session. This means each Claude response generates a separate summary entry. The `MEMSEARCH_NO_WATCH=1` and `CLAUDECODE=` env vars prevent the child `claude -p` process from interfering with the parent session.

**Fallback:** If `claude` CLI is unavailable, raw parsed transcript is appended instead.

## 6. Search Pipeline

**Query flow:**
1. User query -> `MemSearch.search()` -> embed query with configured provider
2. `MilvusStore.search()` creates two `AnnSearchRequest`s:
   - Dense: query embedding against `embedding` field (COSINE)
   - Sparse: query text against `sparse_vector` field (BM25)
3. `hybrid_search()` with `RRFRanker(k=60)` fuses results
4. Returns top_k results with content, source, heading, score, line numbers

**Progressive disclosure (in memory-recall skill subagent):**
- **L1 (search):** `memsearch search "<query>" --top-k 5 --json-output` — chunk-level results
- **L2 (expand):** `memsearch expand <chunk_hash>` — reads original source file, extracts full heading section around the chunk, parses anchor comments for session/turn/transcript metadata
- **L3 (transcript):** `memsearch transcript <jsonl_path> --turn <uuid> --context 3` — parses Claude Code JSONL transcript, shows original conversation turns

**Cold-start context (non-search):** SessionStart hook injects last 30 lines from the 2 most recent daily logs as `additionalContext`. This is raw tail, not semantic search — gives Claude awareness of recent history so it knows when to invoke the skill.

**No search-on-prompt in the hook itself.** The `UserPromptSubmit` hook only sends a lightweight hint. Actual retrieval is pull-based via the memory-recall skill, which Claude invokes autonomously.

## 7. Tool Surface

### CLI Commands
| Command | Description |
|---------|-------------|
| `memsearch index <paths> [--force]` | Scan and embed markdown files |
| `memsearch search <query> [--top-k N] [--json-output]` | Hybrid semantic + keyword search |
| `memsearch watch <paths> [--debounce-ms N]` | Background file watcher with auto-reindex |
| `memsearch compact [--llm-provider] [--source]` | LLM-powered chunk compression to markdown |
| `memsearch expand <chunk_hash> [--section/--lines N]` | Show full context around a chunk (L2) |
| `memsearch transcript <jsonl> [--turn UUID] [--context N]` | Parse JSONL transcript (L3) |
| `memsearch stats` | Show chunk count |
| `memsearch reset --yes` | Drop collection |
| `memsearch config init [--project]` | Interactive config wizard |
| `memsearch config set <key> <value>` | Set config value |
| `memsearch config get <key>` | Get resolved config value |
| `memsearch config list [--resolved/--global/--project]` | Show configuration |

### Python API
- `MemSearch(paths, embedding_provider, milvus_uri, collection, ...)` — constructor
- `await mem.index(force=False)` — index all files, GC stale chunks
- `await mem.search(query, top_k=10)` — hybrid search
- `await mem.compact(source, llm_provider, ...)` — LLM compression
- `mem.watch(on_event, debounce_ms)` — start file watcher
- `mem.close()` — release resources

### Plugin Hooks (4)
SessionStart, UserPromptSubmit, Stop, SessionEnd

### Plugin Skills (1)
memory-recall (context: fork, allowed-tools: Bash)

## 8. Comparison to claude-memory

| Dimension | memsearch | claude-memory |
|-----------|-----------|---------------|
| **Integration** | Claude Code plugin (hooks + skill + CLI) | MCP server (stdio transport) |
| **Storage source of truth** | Markdown files — human-readable, git-friendly | JSON file (`memories.json`) synced to/from SQLite+FAISS |
| **Human editability** | First-class: edit `.md` files, re-index | Indirect: JSON sync layer allows manual editing, but format is structured JSON not prose |
| **Vector backend** | Milvus (Lite/Server/Cloud) with built-in BM25 Function | FAISS (in-process, local only) |
| **Search method** | Hybrid RRF (dense cosine + BM25 sparse) via Milvus native | Custom RRF (FAISS cosine + BM25 with manual TF-IDF), novelty operator for edge dedup |
| **Embedding model** | Pluggable (5 providers), default OpenAI 1536d | Fixed: OpenAI text-embedding-3-small 1536d |
| **Dedup** | SHA-256 content hash as composite PK (source+lines+content+model) | Edge-level dedup with novelty operator (cosine similarity threshold) |
| **Memory decay** | None | Per-category decay with floor (strength field, access_count, half-lives) |
| **Session capture** | Stop hook: extract last turn, haiku summary, append markdown | No auto-session-summary; explicit `remember` tool |
| **Session cold-start** | SessionStart injects tail of 2 recent daily logs | `startup_load` with token-counted budget injection |
| **Consolidation** | `compact` command: LLM summarization to markdown | NREM (dedup/merge) + REM (pattern extraction) sleep cycles |
| **Recall feedback** | None | `recall_feedback` tool for relevance reinforcement |
| **Progressive disclosure** | 3-layer: search -> expand -> transcript (in forked subagent) | Single-layer: search returns results directly |
| **Context isolation** | Forked subagent keeps retrieval out of main context | MCP tools in main context (tool definitions + results consume tokens) |
| **Rebuild from source** | `memsearch reset && memsearch index` — trivial rebuild from .md | `memories.json` -> re-embed; but graph structure (edges, categories, strength) not in markdown |
| **Multi-project** | Per-project Milvus collections (auto-derived from path) | Single memory store (no project isolation) |
| **Windows support** | No Milvus Lite; requires remote Milvus Server or WSL2 | Full Windows support (FAISS, SQLite all local) |

### Key Architectural Differences

1. **Markdown-first vs DB-first:** memsearch's markdown files are genuinely the source of truth — everything derives from them. claude-memory's source of truth is the JSON sync layer, which is a structured format (not prose), though it is human-editable.

2. **Passive capture vs active tools:** memsearch captures memories automatically via Stop hook (every response gets summarized). claude-memory requires explicit `remember` calls — the agent must decide what's worth remembering. memsearch's approach guarantees no information loss but generates more noise; claude-memory's approach is more selective but depends on the agent's judgment.

3. **Retrieval architecture:** memsearch's forked subagent is genuinely clever — it isolates the multi-step retrieval process (search + evaluate + expand + optional transcript drill-down) in a separate context window. Only the curated summary enters the main context. claude-memory's MCP tools operate in the main context, so every search result and tool call consumes tokens.

4. **Memory intelligence:** claude-memory has significantly more sophisticated memory management: decay with per-category half-lives, edge-level novelty dedup, NREM/REM consolidation cycles, recall feedback loops, importance scoring. memsearch's memory management is simple: SHA-256 dedup, no decay, no importance ranking, optional LLM compaction.

5. **Search sophistication:** Both use RRF hybrid search. memsearch delegates to Milvus's native `hybrid_search()` with BM25 Function — battle-tested infrastructure. claude-memory implements its own RRF with manual BM25 scoring plus a novelty operator for edge dedup during retrieval.

## 9. Worth Stealing

**Ranked by value:**

1. **Forked subagent for retrieval (HIGH).** The `context: fork` skill pattern is the single best idea in memsearch. Multi-step retrieval (search + evaluate + expand) happens in isolation; only curated results enter main context. This is architecturally superior to MCP tools consuming main context tokens. claude-memory could implement this by wrapping recall in a skill with `context: fork` instead of exposing raw MCP tools.

2. **Progressive disclosure (L1/L2/L3) (HIGH).** The three-layer drill-down — chunk summary -> full section -> original transcript — is an elegant way to give the agent control over retrieval depth. claude-memory could add an `expand` tool that returns surrounding context for a memory, and potentially link back to session transcripts.

3. **Markdown-as-source-of-truth philosophy (MEDIUM).** The guarantee that you can `rm -rf` the vector index and rebuild from human-readable files is psychologically valuable and practically useful. claude-memory's JSON sync layer achieves something similar but in a less natural format. Worth considering whether the JSON sync could also generate/consume markdown for human readability.

4. **Automatic session capture via Stop hook (MEDIUM).** Every response gets summarized automatically — no agent judgment needed about what to remember. Reduces the "forgot to save" failure mode. The haiku-based third-person summarization is cheap and effective. However, this generates more memories than selective remembering, which makes decay/consolidation more important.

5. **Per-project collection isolation (LOW-MEDIUM).** The `derive-collection.sh` pattern (sanitized basename + path hash) cleanly isolates projects. claude-memory currently has a single memory store. If multi-project support is ever needed, this naming scheme is worth copying.

6. **File watcher for live sync (LOW).** The watchdog-based watcher with debounce keeps the index in sync without manual intervention. Nice for the markdown-first model but less relevant for claude-memory's DB-first approach where writes go directly to the store.

## 10. Not Worth It

1. **Milvus dependency.** Milvus is powerful but heavy — a full vector database for what is essentially a personal memory store. The no-Windows-support for Milvus Lite is a real limitation. FAISS is simpler and sufficient for personal-scale memory (~thousands of entries, not millions).

2. **Shell-script hook architecture.** Bash scripts with inline Python, jq fallbacks, PID file management — this is fragile plumbing. The recursion guard (`stop_hook_active`), the `MEMSEARCH_NO_WATCH=1` / `CLAUDECODE=` env var hacks, the orphan process sweeping — all evidence of accidental complexity from the shell-based approach. MCP's stdio transport is cleaner.

3. **No memory decay or importance scoring.** memsearch treats all memories equally. A session from 3 months ago has the same retrieval weight as yesterday's. claude-memory's per-category decay with floor is strictly better for long-running use. Without decay, the compact command becomes the only way to manage growing memory — and it requires manual invocation.

4. **No recall feedback.** No mechanism to reinforce useful memories or suppress irrelevant ones. claude-memory's `recall_feedback` tool creates a learning loop that improves retrieval quality over time.

5. **Last-turn-only summarization.** The Stop hook only captures the last turn of each response, not the full session. For multi-turn sessions with important context in earlier turns, information is lost unless those turns also triggered Stop hooks. This is by design (cheapness) but limits capture fidelity.

6. **External claude CLI dependency for summarization.** The Stop hook shells out to `claude -p --model haiku`. If the claude CLI isn't installed or the API is down, summarization silently falls back to raw transcript. This is an external dependency that can fail silently.

7. **Plugin-only integration.** The Claude Code plugin system is less portable than MCP. MCP servers can be used by any client (Cursor, other editors, custom agents). The memsearch library itself is framework-agnostic, but the capture/retrieval automation is plugin-locked.
